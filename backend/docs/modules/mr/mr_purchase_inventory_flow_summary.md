# Legacy Analysis Phase 2F：MR 採購與庫存異動核心流程總整理

基於前期的多個模組深度分析，我們已經初步解開了舊系統 (Legacy PB) 的 MR 模組最核心的「採購 -> 入庫 -> 發料 -> 退貨」資料鏈。本文件將整理這些作業的資料流、表關聯與高風險的底層發現。

---

## 一、MR 核心資料流總覽

- **MR035 / MR036**：料號主檔 (Materials) 與供應商報價 (Quotations)。所有採購與庫存皆依賴 MR035 的料號。
- **MR045 / MR046**：採購單主檔與明細。發起採購需求，並由供應商交貨。
- **MR105 / MR106 (`mtype='1'`)**：樣品入庫單。接收來自 MR045 的明細進行入庫，並增加 MR035 庫存。
- **MR110 / MR106 (`mtype='2'`)**：樣品發料單。將已入庫的材料發給特定樣品開發 (`dp030`)，並扣減 MR035 庫存。
- **MR115 / MR106 (`mtype='3'`)**：退貨單。針對不良品，從 MR105 入庫單轉出退貨，扣減庫存，並反推採購單。
- **MR038**：庫存異動明細表 (Inventory Transaction Log)。所有涉及庫存加減的動作 (入庫、發料、退貨)，皆寫入此表。

---

## 二、核心資料表關係

| 資料表 | 定位與說明 | 關聯外鍵與流向 |
| --- | --- | --- |
| **mr035** | 材料/料號主檔 (Master) | `tstdqty` (總庫存) 與 `mprice` (平均成本) 的核心載體。被所有表參照。 |
| **mr036** | 供應商材料報價 | 待確認 (尚未深度分析)。 |
| **mr045** | 採購單主檔 (PO Header) | 參照 `ba015` (供應商)。被 `mr106.importgkey` (入庫明細) 參照。 |
| **mr046** | 採購單明細 (PO Detail) | `mr045gkey` (主檔)，`mr035gkey` (料號)。記錄採購量 (`quantity`) 與已入庫量 (`priceqty`)。 |
| **mr105** | 入庫/發料/退貨主檔 | 透過 `mtype` 區分單據類型。參照 `ba015` (供應商) 或 `dp030` (樣品，僅發料)。 |
| **mr106** | 入庫/發料/退貨明細 | `mr105gkey` (主檔)，`mr035gkey` (料號)，`importgkey` (來源單據，對應 `mr046` 或前手 `mr106`)。 |

| **es101** | 員工主檔 | 建檔人、審核人、領料人。 |

## 三、mtype 語意整理

舊系統將入庫、發料、退貨高度濃縮在同一張表 (`mr105` / `mr106`) 中，利用 **`mtype`** 來控制不同的商業邏輯：

| mtype | 對應作業 | mr035.tstdqty | mr035.mprice | mr038.mtype | 回推來源單據 |
| --- | --- | --- | --- | --- | --- |
| **'1'** | **MR105 入庫** | **增加** (+) | 重新計算移動平均成本 | **'1'** | 回寫 `mr046`, `mr045` |
| **'2'** | **MR110 發料** | **扣減** (-) | 讀取目前成本，不重新計算 | **'2'** | 無 (純粹出庫) |
| **'3'** | **MR115 退貨** | **扣減** (-) | 啟動公式反向計算平均成本 | **'3'** | 回寫並竄改 `mr106` (入庫)<br>重啟 `mr046`, `mr045` |

## 四、MR105 入庫流程

- **來源**：轉單自未交齊 (`status <> '4', '3'`) 的 MR045/MR046 採購明細。
- **過帳條件**：使用者點擊審核 (`cfmchk='Y'`) 即時觸發過帳。
- **盤點阻擋**：若系統參數 `stocktaking='T'`，阻擋入庫。
- **庫存增加邏輯**：`mr035.tstdqty = mr035.tstdqty + mr106.changeqty`。
- **移動平均成本**：依據系統參數，使用 `((原庫存量 * 原單價) + (本次入庫量 * 採購單價)) / (原庫存量 + 本次入庫量)` 計算新單價。
- **mr038 寫入**：新增一筆紀錄。
- **來源單據回寫**：
  - 更新 `mr046.priceqty = mr046.priceqty + 入庫量`。
  - 若 `priceqty >= quantity`，則 `mr046.chk = 'Y'` (明細交完)。
  - 更新 `mr045.status` ('2' 入庫中 或 '3' 入庫完成) 與 `mr045.pamount` (累計請款金額)。

---

## 五、MR110 發料流程

- **來源**：可轉單自已審核的 MR105 入庫單 (`mtype='1'`)，或手動 F2 選擇料號。
- **過帳條件**：使用者點擊審核 (`cfmchk='Y'`) 即時觸發過帳。
- **負庫存防呆**：發料量不可大於 `mr035` 目前庫存，若不足則標記紅色並阻擋過帳。
- **庫存扣減邏輯**：`mr035.tstdqty = mr035.tstdqty - mr106.changeqty`。
- **移動平均成本**：不重新計算。直接沿用 `mr035.mprice` 作為出庫成本。
- **mr038 寫入**：新增一筆紀錄。
- **來源單據回寫**：無。發料不會回推更新 `dp030`，單純作為出庫憑證。

---

## 六、MR115 退貨流程

- **來源**：嚴格轉單自已審核的 MR105 入庫單明細。
- **付款狀態阻擋**：如果原採購單已經付款 (`mr045.status='4'`)，嚴格阻擋退貨。
- **負庫存防呆**：退貨量不可大於「原入庫量」，且扣減後庫存不可為負。
- **庫存扣減邏輯**：`mr035.tstdqty = mr035.tstdqty - 退貨量`。
- **移動平均成本 (反算)**：使用公式 `((原庫存量 * 原單價) - (退貨量 * 採購單價)) / (原庫存量 - 退貨量)` 把退貨前的平均單價反算回來。
- **mr038 寫入**：新增一筆紀錄。
- **連鎖回扣 (Chain Reversal)**：
  - 直接竄改原 MR105：`mr106(入庫).changeqty -= 退貨量`。
  - 回扣原 MR046：`mr046.priceqty -= 退貨計價量`。如果扣完後未交齊，將 `mr046.chk` 重啟為 `'N'`。
  - 重啟原 MR045：將 `mr045.status` 倒退為 `'2'`(入庫中) 或 `'1'`(採購中)。

## 七、已發現的高風險底層邏輯

1. **歷史單據 Hard Update**：MR115 退貨並不是產生獨立的紅字沖帳單，而是**直接去 update (竄改) 原先已經審核的 MR105 入庫單明細**。這在資料庫一致性上有極高風險，必須嚴格以 DB Transaction 包覆。
2. **Moving Average Cost 正反算**：入庫時正算成本，退料時反算成本。如果期間有其他入庫或發料發生，退料的反算邏輯極易導致小數點誤差或單價失真，需在實作 Service 時小心處理精確度 (Decimals)。
3. **付款狀態阻擋**：庫存異動 (MR115) 竟然與財務狀態 (`mr045.status='4'`) 強綁定。
4. **盤點狀態阻擋**：系統全域參數 `stocktaking='T'` 會鎖死所有庫存異動。
5. **負庫存防呆**：舊系統預設禁止發料與退貨導致負庫存。
6. **審核即過帳**：所有 PB Window 都將「存檔審核」與「扣庫存過帳」綁在一起 (`ue_updatestock`)。
7. **反審核 / 取消過帳 (Unapprove)**：目前分析未明確發現取消審核的邏輯（退料似乎被當作取消入庫的替代方案），仍待確認。
8. **Transaction 依賴**：這四大單據牽扯多張 Table 更新，所有過帳邏輯必須包裝在 `transaction.atomic()` 內。

## 八、未來底層 Service 需求 (開發參考)

根據目前的架構，未來必須實作以下核心 Service，**絕不能寫在 View / Serializer 內**：

- **`InventoryPostingService`**：統一處理所有 `mr035` 庫存加減、移動平均單價計算，以及自動寫入 `mr038`。
- **`PurchaseOrderReceiveService`**：負責 MR105 入庫時，累加 `mr046` 數量與推進 `mr045` 狀態的邏輯。
- **`PurchaseOrderReversalService`**：專門處理 MR115 這種連鎖回推 (Chain Reversal) 與狀態重啟的破壞性邏輯。
- **`SystemParameterService`**：提供 `stocktaking` (盤點中)、`averagecost` (是否計算平均成本) 等全域狀態的快取查詢。
- **`PostingGuard`** / **`DeleteGuard`**：阻擋負庫存、阻擋已付款單據異動、阻擋盤點期間異動。

## 九、後續待分析事項

雖然核心採購/庫存流已梳理清楚，但仍有以下未知領域：

1. **MR050 / MR047 (財務與請款)**：如何讀取 `mr045` 的請款資料？請款是否會改變 `mr035` 的入庫成本 (補差價)？
2. **MR053 (請購單/詢價單)**：這張單據如何轉入 MR045？
3. **SM (成品/大貨庫存模組)**：大貨的庫存是否共用 `mr035` / `mr038`，還是另外有一套 `sm` 開頭的庫存表？
4. **mr038 完整欄位語意**：`mr038` 除了 `qty`、`nqty`，是否還有紀錄批號、倉庫等資訊？
5. **反審核邏輯**：舊系統是否有 `Unapprove` 按鈕？如果有，反向更新該如何運作？

## 十、輸出結論

1. **核心鏈條已初步解開**：從 MR045 -> MR105 -> MR110 / MR115 的這條「樣品材料採購與庫存」主幹已經清晰，資料流向與 Table 依賴皆已掌握。
2. **必須共用 Model 與 Service**：MR105、MR110、MR115 **必須**共用同一個 `Mr105` Model 與 `Mr106` Detail Model，並且強烈依賴 `InventoryPostingService`。不能將這三支作業分開給不同人以不同標準實作。
3. **後續分析建議**：
   - 往 **財務流** 前進：分析 MR047 / MR050，釐清 `mr045.status='4'` 是如何產生的。
   - 往 **源頭流** 前進：分析 MR036 / MR053，釐清材料報價與請購邏輯。
   - 往 **大貨庫存** 橫向探索：檢查 SM 模組是否與 MR 模組底層打架。
