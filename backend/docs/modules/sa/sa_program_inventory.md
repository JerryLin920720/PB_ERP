# Legacy Analysis Phase 1：SA 業務部門管理系統作業總盤點

## 1. SA 模組總覽

- **SA 模組定位**：鞋貿 ERP 的「業務部門管理系統」（Sales Department Management System），負責處理客戶報價、樣品訂單、預告訂單（Forecast）、正式訂單（P/O）、與形式發票（P/I）的建立與管理，同時提供業務相關的統計查詢與報表。
- **業務部門主要負責範圍**：對外與客戶溝通接單、報價，對內發行開模通知、預告訂單與正式訂單，協調 DP（開發）與 MR（資材）、SM（出貨）確保訂單能按時生產與交付。
- **與其他模組關聯概述**：
  - **BA (基礎設定)**：依賴客戶資料、幣別、付款條件等。
  - **DP (開發)**：依賴型體 (dp025)、楦頭 (dp010)、大底 (dp015)、鞋跟 (dp020) 等基礎開發資料，訂單變更會影響 DP 的 adopted 狀態。
  - **MR (資材)**：依賴材料與廠商資訊，訂單確立後拋轉計算物料需求。
  - **SM (出貨)**：訂單資料為出貨 (Shipping) 的來源依據。
  - **FA (財務)**：P/I (形式發票) 與 P/O (正式訂單) 金額作為應收帳款與收款的依據。

## 3. Pattern 分類統計

* **Pattern A (單表 CRUD / 基本設定)**：4 支 (SA001, SA005, SA006, SA007)
* **Pattern A2 (上列表下編輯)**：2 支 (SA046, SA095)
* **Pattern B (Master-Detail 工作台)**：6 支 (SA010, SA012, SA015, SA020, SA030, SA040, SA352 可併為同一大類)
* **Pattern R (查詢報表 / 統計)**：13 支 (SA048, SA050, SA055, SA058, SA060, SA065, SA070, SA075, SA080, SA085, SA090, SA096)
* **Pattern Special (特殊流程 / 批次 / 審核)**：2 支 (SA018, SA045)
* **未判定**：0 支

## 6. 需要深度分析的 SA 作業

在進入實際開發前，下列作業需要進行優先的深度分析：

1. **SA030 (P/O 訂單資料管理)**：SA 最核心作業，包含複雜的 Master (sa030) 與 Detail (sa031, sa032, sa033, sa034, sa035)。涉及裝箱 (Assortment) 與多顏色尺碼的複雜輸入介面，並深度影響 DP 開發資料狀態 (adopted)。
2. **SA020 (預告訂單資料管理)**：與 SA030 幾乎共用資料表 (sa030 狀態不同)，需要與 SA030 一併分析，規劃共用元件。
3. **SA040 (P/I 訂單資料管理)**：依賴 SA030 的資料流，需要確認 SA030 到 SA040 的拋轉與資料關聯。
4. **SA012 / SA010 (報價資料管理)**：業務前端報價作業，包含複雜的報價單列印，並依賴 DP 模組的樣品與 B.O.M 成本資料。

## 7. 依賴與關聯

* **BA (基本資料)**：幾乎所有作業都依賴客戶 (BA010)、工廠、幣別、付款條件、港口等基本設定。
* **DP (開發)**：SA020/SA030 建立時會改變 `dp010` (楦頭)、`dp015` (大底)、`dp020` (鞋跟)、`dp025` (型體) 的 `adopted` 狀態。SA012 (報價) 強烈依賴 DP 的成本與樣品資料。
* **QC (品保)**：SA030 中提到會在 `qc030`, `qc031` 自動插入或刪除檢驗資料，這部分跨模組連動需特別注意。
* **MR (資材)**：訂單建立後會產生資材需求，SA046 配件收受也可能與 MR 關聯。
* **SM (出貨)**：SA030 訂單資料是出貨的根本來源，SA058 未出貨統計依賴 SM 出貨資料。
* **FA (財務)**：SA096 依賴 L/C (信用狀) 資料，P/I (SA040) 依賴 FA 應收帳款體系。

## 8. 第一批建議開發順序

* **Batch 1：低風險，可立即開發**
  * 皆已完成 (SA001, SA005, SA006, SA007)
* **Batch 2：需要 Master-Detail 深度分析 (SA 核心架構)**
  * SA030 (P/O 訂單資料管理)
  * SA020 (預告訂單資料管理) - 可與 SA030 同步進行
* **Batch 3：需要訂單 / 報價 / 改單流程確認**
  * SA012 / SA010 (報價資料管理)
  * SA040 (P/I 訂單資料管理)
  * SA045 (訂單狀態審核管理)
* **Batch 4：需要報表或跨模組依賴**
  * 所有 Pattern R 的統計查詢作業 (SA048 ~ SA096)
  * 等待 BA/DP/FA/SM 資料完備後再進行實作。

## 9. 風險與待確認事項

* **與 QC 模組的隱性連動**：在 `w_sa020` 中發現 `ids_qc030` 的自動新增/刪除邏輯。此類跨模組自動寫入邏輯在重構時應評估是否改為 Event Driven 或交由 QC 模組管理。
* **DP 資料 adopted 狀態維護**：`sa030` 狀態異動會反向更新 `dp010`, `dp015`, `dp020`, `dp025` 的 `adopted`。在 API 設計時需確保分散式更新的 Transaction 安全。
* **SA030 的高複雜性**：包含 Assortment 展開、尺碼網格編輯，這是前端 React 實作的難點，需要特別設計針對 Size / Color 展開的 UI 元件。
* **PB 報表格式遺失風險**：諸如 `d_sa030_report_vk1`, `d_sa012_report` 等大量報表需重新評估是否以 Excel/PDF 產生，需要先請業務提供現行輸出的紙本樣張。
* **SA352 的定位**：看似為 SA030 的客製版，應與使用者確認是否合併為同一支程式靠權限或參數控制，避免維護兩套相似的 Master-Detail。
